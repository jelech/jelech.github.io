简介

增强的Go-template方法用于编译k8s部署yaml

Helm 是 Kubernetes 的包管理器，Helm的包涵盖了将Kubernetes资源安装到Kubernetes集群所需要的足够多的信息。而其中的charts包含Chart.yaml文件和模板，默认值(values.yaml)，以及相关依赖。

helm chart 可以方便编写和部署k8s的应用，方便编写和组织yaml文件。它使用GO的模板引擎为基础，增加了部分的字段，用于Go模板流水线。

  • 增加include方法
  • 增加required方法
  • 增加tpl方法

注意所有的Helm内置变量以大写字母开头，以便与用户定义的value进行区分：.Release.Name，.Capabilities.KubeVersion

Demo

  • helm create app
  • helm install full-coral-name ./app -n namespace
    • -n 表示指定的k8s的该namespace下进行操作
  • helm get manifest full-coral
    • 获取发布的full-coral-name
  • helm uninstall full-coral
    • 卸载发布的程序，改操作同时也会移除线上链接的对应服务
  • helm install --debug --dry-run goodly-guppy ./mychart
    • 不会进行安装到k8s集群，只会渲染最终结果到控制台用于测试
    • --disable-openapi-validation用于查看渲染结果
  • value 覆盖： values.yaml < 父values.yaml < 外部-f提供的value文件 < -set的值
  • value中设置key=null可以删除该value，注意替代顺序
  • {{ .Values.favorite.drink | upper | quote }}
    • 管道 | 可以链接数据进行多个函数调用
    • 已知：upper repeat n quote
    • helmChart函数列表
  • default "value"设置默认值
  • lookup "v1" "Pod" "mynamespace" "mypod" = kubectl get pod mypod -n mynamespace
    • 使用loopup函数进行获取k8s的相关功能
    • (lookup "v1" "Namespace" "" "mynamespace").metadata.annotations使用k8s namespace的特定属性值
  • |-通配符用于表明多行字符串
  • app.kubernetes.io/instance: "{{ $.Release.Name }}"
    • $符号在单独使用的时候可以指代全局作用域
  • define操作允许我们在模板文件中创建一个命名模板
    • {{- template "mychart.labels" }} 进行使用模板
    • {{- template "mychart.labels" . }}传入范围用于内部调用
    • {{ include "mychart.app" . | indent 4 }}Helm模板中使用include而不是template被认为是更好的方式 只是为了更好地处理YAML文档的输出格式
      {{ define "MY.NAME" }}
# body of template here
{{ end }}

    helmfile

    Helmfile 是用于部署 Helm 图表分布的声明性规范包装。它们允许您将多个图表组合在一起以创建一个全面的部署工件，从而为 Helm 添加了额外的功能。

    组织多个helmchart。IAC形式描述多个服务的架构。真正实现一次性可选的部署多个chart工具。同等的工具类似于gitops。

    设置运行时传入的args的默认值。以下为helm的一些默认设置，这些配置与 helm SUBCOMMAND 相同

    helmDefaults:
  tillerNamespace: tiller-namespace  # tiller-namespace的专用默认密钥 （tiller处于k8s服务器中）
  tillerless: false                  # tillerless的专用默认密钥
  kubeContext: kube-context          # kube-context的专用默认密钥 (--kube-context)
  cleanupOnFail: false               # helm flag的专用默认密钥 --cleanup-on-fail
  # 传递给 helm 的附加和全局 args (default "")
  args:
    - "--set k=v"

  # 在upgrade前验证chart（仅适用于打包图表而不是目录）（默认为 false）
  verify: true

  # 通过 --wait 等待 k8s 资源。(默认 false)
  wait: true

  # 如果设置并启用--wait，将等到所有作业完成后才会将发布标记为成功。它将等待--timeout (默认为 false，在 Helm3.5 中实现)
  waitForJobs: true

  # 以秒为单位等待任何单个 k8s 操作的时间（如钩子的作业，并等待 pod/pvc/svc/deployment就绪）（默认 300）
  timeout: 600

  # 如果可以的话，对资源执行 pods 重启（默认 false）
  recreatePods: true

  # 如果需要，通过删除/重新创建强制资源更新（默认 false）
  force: false

  # 为对 Tiller 的请求启用 TLS（默认 false）
  tls: true

  # TLS-CA 证书文件路径 (默认 "$HELM_HOME/ca.pem")
  tlsCACert: "path/to/ca.pem"

  # TLS-Cert 文件的路径 (默认 "$HELM_HOME/cert.pem")
  tlsCert: "path/to/cert.pem"

  # TLS-key 文件的路径 (默认 "$HELM_HOME/key.pem")
  tlsKey: "path/to/key.pem"

  # 每个版本保存的最大修订数。使用 0 表示无限制。(默认 10)
  historyMax: 10

  # 使用 helm 3.2+ 时，如果不存在则自动创建发布命名空间（默认为 true）
  createNamespace: true

	# 如果与charts一起使用，则museum允许提取不稳定charts进行部署，例如：如果 1.2.3 和 1.2.4-dev 版本存在并设置为 true，1.2.4-dev 将被拉取（默认为 false）
  devel: true

  # When set to `true`, skips running `helm dep up` and `helm dep build` on this release's chart.
  # Useful when the chart is broken, like seen in https://github.com/roboll/helmfile/issues/1547
	# 当设置为 `true` 时，跳过在这个版本的chart运行 `helm dep up` 和 `helm dep build`。
  # 当图表损坏时很有用，如 https://github.com/roboll/helmfile/issues/1547
  skipDeps: false

    releases设定: 基础release设定

    releases:
  - name: vault                        # 版本名称
    namespace: vault                   # 目标namespace
    createNamespace: true              # helm 3.2+ 自动创建发布namespace（默认为 true）
    labels:                            # 用于过滤发布的label
      foo: bar
    chart: roboll/vault-secret-manager # 对应的chart referenced by `repository/chart` syntax
    version: ~1.24.1                   # 版本范围约束。支持范围设定
    condition: vault.enabled           # 用于过滤发布的值查找键。对应于 `vault.enabled` 的布尔值，其中 `vault` 是任意值
    missingFileHandler: Warn           # 设置为“错误”或“警告”。 “错误”指示 helmfile 在无法找到值或机密文件时失败。当“警告”时，它会打印文件并继续。 # 用于渲染图表的值文件

    # Values files used for rendering the chart
    values:
      # 通过 --values 传递的值文件，用于覆盖默认的value文件
      - vault.yaml

      # 内联值，通过临时值文件和--values 传递，因此它不会受到--set 等类型问题的影响
      - address: https://vault.example.com

      # Go 模板可在内联值和值文件中使用。
      - image:
          # 最终结果也很可能是yaml。 `quote` 也是如此，以防止类似数字的字符串被意外解析为数字！ 见 https://github.com/roboll/helmfile/issues/608
          tag: {{ requiredEnv "IMAGE_TAG" | quote }}
          # 否则:
          #   tag: "{{ requiredEnv "IMAGE_TAG" }}"
          #   tag: !!string {{ requiredEnv "IMAGE_TAG" }}
        db:
          username: {{ requiredEnv "DB_USERNAME" }}
          password: {{ requiredEnv "DB_PASSWORD" }}
        proxy:
          domain: {{ requiredEnv "PLATFORM_ID" }}.my-domain.com
          scheme: {{ env "SCHEME" | default "https" }}

    release设定：set字段值

    releases:
    # 请尽量使用values
    # `set` 转换为 helm 的 `--set key=val`，已知它会遇到类型问题，例如 https://github.com/roboll/helmfile/issues/608
    set:
    - name: foo.config    # 从本地文件加载的单个值, 转换为 --set-file foo.config=path/to/file
      file: path/to/file
    - name: bar[0]        # 在数组中设置单个数组值，转换为 --set bar[0]={1,2}
      values:
      - 1
      - 2
    - name: namespace     # 设置模板值
      value: {{ .Namespace }

    高级配置：嵌套状态

    helmfiles:
- # 在发布releases前的helmfile文件路径
  path: path/to/subhelmfile.yaml
  
	# 标签选择器用于过滤嵌套状态的发布
  # 比如这个上下文中的`name=prometheus`等价于处理嵌套状态如 # helmfile -f path/to/subhelmfile.yaml -l name=prometheus sync
  selectors:
  - name=prometheus

  # 覆盖状态值
  values:
  - additional.values.yaml # 值文件合并到嵌套状态的值中
  # 这里使用值的一个重要方面是它们首先需要在原始 helmfile 的 values 部分中定义，因此在此示例中 key1 需要位于 path/to/subhelmfile.yaml 的 values 或 environment.NAME.values 中内联状态值合并到嵌套状态的值中
  - key1: val1

- # `helmfiles:` 下的所有嵌套状态文件按定义顺序处理。所以它可以用来准备你的主要`releases`。一个例子是在父状态文件中创建 `releases` 所需的 CRD。
  path: path/to/mycrd.helmfile.yaml

- # Terraform-module-like URL 用于导入远程目录并将其中的文件用作嵌套状态文件，嵌套状态文件与包含它的远程目录一起在本地签出。因此文件中的所有本地路径都是相对于文件
  path: git::https://github.com/cloudposse/helmfiles.git@releases/kiam.yaml?ref=0.40.0
